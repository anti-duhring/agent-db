    2. Create `internal/benchmark/runner.go`:
       - Package `benchmark`
       - Import `hdrhistogram "github.com/HdrHistogram/hdrhistogram-go"`
       - Define `RunConfig` struct per D-08:
         ```go
         type RunConfig struct {
             Backend     string
             Warmup      int
             Iterations  int
             Concurrency int
             Profile     generator.Profile
             Seed        int64
         }
         ```
       - Define `Runner` struct:
         ```go
         type Runner struct {
             repo      repository.ChatRepository
             scenarios []Scenario
             config    RunConfig
         }
         ```
       - `NewRunner(repo repository.ChatRepository, scenarios []Scenario, config RunConfig) *Runner`
       - `func seedRepository(ctx context.Context, repo repository.ChatRepository, data generator.GeneratedData) error`:
         - Iterate `data.Conversations`, call `repo.CreateConversation(ctx, c.PartnerID, c.UserID)` for each — but NOTE: CreateConversation generates its own ID. Instead, use the repo interface directly. Since the interface does not accept an ID, the seed data conversations will have different IDs than the generated ones. To work around this:
           - Call `repo.CreateConversation(ctx, c.PartnerID, c.UserID)` which returns a `domain.Conversation` with the DB-assigned ID
           - Then iterate `data.Messages[c.ID]` and call `repo.AppendMessage(ctx, createdConv.ID, msg.Role, msg.Content)` using the DB-assigned conversation ID
         - This ensures FK integrity. The returned conversations from CreateConversation should be collected and returned so scenarios know the actual DB conversation IDs.
       - Adjust `seedRepository` signature to: `func seedRepository(ctx context.Context, repo repository.ChatRepository, data generator.GeneratedData) ([]domain.Conversation, error)` — returns the actual DB conversations
       - Define `SeedResult` struct to pass to scenarios:
         ```go
         type SeedResult struct {
             PartnerID     uuid.UUID
             UserID        uuid.UUID
             Conversations []domain.Conversation
             OriginalData  generator.GeneratedData
         }
         ```
       - Update `Scenario.Setup` in scenario.go to accept `SeedResult` instead of `generator.GeneratedData`:
         ```go
         Setup(ctx context.Context, repo repository.ChatRepository, seed SeedResult) error
         ```
       - `func (r *Runner) Run(ctx context.Context) ([]ScenarioResult, error)`:
         - Create generator: `gen := generator.New(r.config.Seed)`
         - Generate deterministic partner/user IDs: use the generator's seeded faker for these, OR use fixed UUIDs. Simplest: use `uuid.New()` for partnerID and userID (these don't need to be deterministic across runs — only the data content does).
         - Generate data: `data := gen.Generate(partnerID, userID, r.config.Profile)`
         - Seed repository: `convs, err := seedRepository(ctx, r.repo, data)`
         - Build `SeedResult{PartnerID: partnerID, UserID: userID, Conversations: convs, OriginalData: data}`
         - For each scenario:
           a. Call `sc.Setup(ctx, r.repo, seedResult)`
           b. Create histogram: `h := hdrhistogram.New(1, 30_000_000, 3)` (1us to 30s, 3 sig digits)
           c. Warmup loop (per D-08): run `r.config.Warmup` iterations calling `sc.Run(ctx, r.repo)`, discard results, log errors but don't fail
           d. Measured loop: run `r.config.Iterations` iterations:
              - `start := time.Now()`
              - `err := sc.Run(ctx, r.repo)`
              - `elapsed := time.Since(start).Microseconds()`
              - If err != nil, return error
              - `h.RecordValue(elapsed)` — log warning if RecordValue returns error (out of range)
           e. Call `sc.Teardown(ctx)`
           f. Append `ScenarioResult{ScenarioName: sc.Name(), P50: h.ValueAtPercentile(50.0), P95: h.ValueAtPercentile(95.0), P99: h.ValueAtPercentile(99.0), TotalCount: h.TotalCount()}`
         - Return results slice
  </action>
